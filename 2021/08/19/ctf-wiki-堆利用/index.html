堆中的 Off-By-One

asis2016_b00ks

查看程序基本信息：

x86_64，little-endian。因为已经知道题目的考点是off-by-one，所以就直接分析。

程序基本功能：

可以创建book、删除book、编辑book、打印book详细信息、修改author name。

程序首先会要用户输入author name，author name的大小限制为32byte。author name是一个全局指针，指向存放author name的空间，大小就是32bytes。

在处理用户输入的author name的self_read（sub_9F5）函数中，存在off-by-one漏洞：

这里的10是换行符’\n’的ascii。self_read函数首先会读取第一个字符，判断是否为NULL，如果不为NULL，就读取内容。self_read函数会读取字符串结尾的’\n’，并将其转换成\x00，这个\x00就是溢出的one byte。虽然程序限制了author name最大为32 bytes，但是字符串最后的\x00总是被读取的，所以最多会读入33 bytes。这就导致最后的\x00会覆盖author name空间的下一个byte的空间。

Create：

Create模块接收用户输入的book name和book description，然后保存到book结构体中。在处理用户输入的name和description的时候，也用到了存在漏洞的self_read函数。

Create模块还需要关注的一个地方就是每一个book的内容是存放到一个结构体中的。

还有一个off_202010+v2其实是存放的book struct的首地址的全局变量（数组）。通过仔细观察可以发现off_202010这个全局变量正好是在author name这个全局变量的相邻低地址处，它们俩指向的地址空间也是相邻的。所以可以通过author name的off-by-one漏洞覆盖第一个book的地址的低字节为\x00。book_ptr+6是因为dword，这里是ida识别的问题，转换成qword就是book_ptr+3了。

这里就出现漏洞的第一个利用点——泄露book的地址。author name最后的\x00会覆盖book指针的低地址（低地址是因为小端存储），但是当Create第一个book时，book的地址又会将\x00覆盖掉，由于没有\x00，打印author name的时候就可以泄露book的地址。

这个题的突破点就是Change current author name模块，可以再覆盖第一个book chunk地址的低字节为\x00。流程是在Create book1 chunk之后Change author name，然后利用off-by-one漏洞就可以覆盖book1 chunk的地址。

book1 chunk的地址的低1byte覆盖成00。

继续分析程序；

edit_book函数可以修改description的内容，程序根据book结构体的book_description_ptr找到要修改的description的地址空间，并向里面写入修改后的description。如果能够控制book_description_ptr的内容，就可以利用edit_book函数实现任意地址写。同样的道理，print_book函数打印book的description的内容时，也是根据book结构体中的book_description_ptr来找description的，所以控制了book_description_ptr，就可以实现任意地址读。总之，控制book_description_ptr，可以实现任意地址读写。

现在不能直接控制book_description_ptr的值，只是可以修改book1 chunk的地址为0xXXXXXXXX00。这里可以间接控制book_description_ptr的值，具体步骤是先伪造一个fake book，然后覆盖book1的指针使其指向fake book。当然，fake book的起始地址需要满足和book1的地址的最低一个字节覆盖为\x00之后的值相等。可能会想，将book1的地址覆盖成\x00结尾不是将地址变小了吗，为什么还能指向fake book，这里程序设计的比较好的地方是程序先申请book_name chunk，然后再申请book_description chunk，最后再申请book struct chunk，所以book chunk在高地址，地址减小之后可以落在book_name chunk或者book_description chunk中。所以自然就会想到将fake book放在book_name chunk或者book_description chunk中。虽然程序开启了地址随机化，但是只有地址的高位会变化，低3位一般不会变。

所以通过前面的分析，这个题目的主要的利用过程是：

  1. 首先准备两个book，book1来实现任意地址读写，book2来实现泄露libc基地址。
  2. 在book1的description中伪造一个fake_book1_struct，通过off-by-one漏洞覆盖book数组中book1_struct首地址的低字节，使原先的book1_struct变成伪造fake_book1_struct。在fake_book1_struct中通过构造book_description_ptr的值，然后调用edit和print函数实现任意地址读写，前提是malloc的book_description要足够大，才能让book1_struct指针修改后落在里面。
  3. Create book2的时候，malloc book_name和book_description的时候大小要非常大（如0x21000），malloc函数才会调用mmap来开辟堆空间，mmap的堆内存都存在于mmap段中，libc也在mmap段中，所以得到book2中book_name_ptr和book_description_ptr，就可以通过偏移得到libc基址。
  4. 结合2,3两点，在伪造fake_book1_struct，可以让book_description_ptr指向book2的book_name_ptr，然后就可以通过edit book1修改book2的内容。因为调用edit函数的时候，修改的是book中book_description_ptr所指向的内容。

Create book2主要是为了后面利用的过程比较方便。

为了方便调试，可以临时禁用系统的地址随机化功能：echo 0 > /proc/sys/kernel/randomize_va_space。注意要切换到root用户执行命令，sudo会报错。

首先Create book1和book2，然后打印，泄露出book1_struct和book2_struct的地址。

构造fake_book1_struct，book1_struct的地址0x555555758490，book1的description的地址是0x555555758380，当覆盖book1_struct的地址为0x555555758400，就会落在description中，伪造的fake_book1_struct的起始地址就是0x555555758400。fake_book1_struct的具体内容是id=1，book_name_ptr赋一个合法地址就可以，book_description_ptr=book2_name_ptr，book_size赋一个合法的大小就可以。

修改author_name，覆盖book1_struct的地址。

泄露libc地址。因为book2的description在mmap段中，和libc基址的偏移不变，vmmap查看libc基地址，并泄露book2_description_ptr，可以得到libc基址和description的地址的偏移，利用偏移和泄露的book2_description_ptr就可以得到远程libc基址。

偏移：0x7ffff7fbd010-0x7ffff79e2000=0x5db010。

getshell。由于程序启用了FULL RELRO保护措施，无法对GOT进行改写，但是可以改写__free_hook或__malloc_hook。 通过前面计算的libc基址，可以得到__free_hook和system的地址，以及libc中”/bin/sh”字符串的地址。将__free_hook指向的内容修改为system的地址，接下来调用free的时候，__free_hook的值不为null，就会执行__free_hook指向的内容（system）。system函数的参数就是要free的chunk中的内容。

因为book1_description_ptr是指向的book2_name_ptr的，所以修改book1就是修改book2_name_ptr和book2_description_ptr的值，将book2_name_ptr修改为”/bin/sh”的地址，book2_description_ptr修改为__free_hook的地址，这样在修改book2的时候修改的是book2_description_ptr所指向的值，也就是__free_hook的值，将其修改为system之后，在free book2时，会执行system，并将book2_name的内容，也就是”/bin/sh”作为system的参数，最后getshell。

Chunk Extend and Overlapping

HITCON Trainging lab13

保护：

只开启了Canary和NX保护。

架构：x86_64

程序实现的是一个堆管理器：

主要功能：

  1. Create a Heap：

    最多可以创建10个heap，首先申请0x10大小的空间来存放heap的size和content地址，然后用户输入heap的size和content，程序申请size大小的空间来存放content；所有的heap通过heaparray的数组结构来管理；

  2. Edit a Heap：

    用户输入需要修改的heap的索引index，然后判断index是否合法，最后根据index可以修改对应的heap中的content；因为在读入用户输入的content的时候对读取了一个字节*(_QWORD *)*(&heaparray + v1) + 1LL，存在off-by-one漏洞；

  3. Show a Heap：

    根据用户输入的index显示heap的size和content；

  4. Delete a Heap：

    根据用户输入的index删除heap，释放申请的content和heap的堆空间，并置为NULL。

漏洞利用：

  1. 通过off-by-one漏洞可以覆盖相邻的下一个chunk的size的低字节，修改下一个chunk的大小；
  2. 申请修改后的chunk，产生chunk overlapping，修改关键指针实现任意地址读写；
  3. 通过修改content的内容来泄露free函数的got表地址，进而计算出libc基地址，然后修改free函数的got表为system函数的地址，最后free触发system函数。

2015 hacklu bookstore

保护：

开启了Canary和NX保护。

架构：x86_64

主要功能：

程序是一个书店的订书系统，每次最多可以订两本书。

程序一开始就malloc了order1，order2和dest的空间，用户可以编辑和删除order，然后是malloc一个0x140大小的堆块并提交order，最后是打印order的信息。在最后printf输出的地方存在格式化字符串漏洞。

edit_order：

edit_order没有长度限制，存在溢出，可以一直接收输入直到输入\n字符。

delete_order：

delete_order函数中free堆块，free之后没有将堆块置为NULL，存在UAF。

submit：

submit函数是将order1和order2的内容通过strcat和strncat连接起来。因为order1和order2存在溢出，所以在submit中也存在溢出；

通过溢出漏洞和UAF可以布置堆空间，伪造chunk；格式化字符串可以实现任意地址读写。程序在submit之后就直接退出了，而格式化字符串是在submit之后才出现的，因为不能预先知道system或者execve函数的地址，所以要先利用格式化字符串泄露地址，然后再利用格式化字符串写指定的地址，所以格式化字符串要利于两次，要让程序在main函数运行结束后再返回main函数执行。这里不能覆盖main函数的返回地址，因为堆溢出覆盖不到栈地址，而且不知道栈上的返回地址是多少，所以覆盖main函数的返回地址行不通。另外一种改变程序执行流程的方法是修改 .fini_array 数组，程序的执行流程：

具体参考：https://xuanxuanblingbling.github.io/ctf/pwn/2019/09/06/317/

可以在IDA中Ctrl+s查看程序的.fini_array段：

.fini_array[]数组中只有一个数据项，可以通过格式化字符串修改.fini_array[0]中的内容为main函数的地址，就可以让main函数再运行一次，注意修改.fini_array只能执行一次main函数，这很好理解，因为程序只执行.fini_array[0]一次，将.fini_array[0]中的内容换成main函数的地址只是再执行一次main函数，执行完第二次main函数之后就执行.fini段中的代码。在第二次执行main函数的时候，才可以利用格式化字符串写第二次main函数的返回地址为system/execve/gadget的地址；

要利用格式化字符串实现任意地址读写，需要满足两个条件，一是需要先将要读写的内容的地址放到栈上，在main函数中读取用户输入的选项时最多可以读取128bytes，可以将需要读写的地址通过这里的fgets函数读取到栈上，然后再利用格式化字符串漏洞实现任意地址读写；

二是需要构造格式化字符串并将它作为printf函数的参数，格式化字符串根据需要来构造，printf函数的参数是dest，所以需要将格式化字符串放入dest中；

因为dest是堆块，所以可以用之前的堆溢出向dest中写入格式化字符串；

但是有一个问题，如果直接利用edit_order1或者edit_order2的溢出功能修改dest中的内容之后，会有一个strcpy操作，会将之前写入dest的格式化字符串覆盖掉，所以不能在edit_order中向dest中写东西；

修改dest的内容除了上面说的代码之外，在submit中也可以修改dest。

之前分析在submit中也存在堆溢出，但是因为submit中使用的堆块(v5)是在dest之后申请的，所以有溢出也不能覆盖到dest中，这里需要想办法让后面malloc的0x140的堆块在dest之前，做法有两种，一种是先通过order1 chunk的溢出修改order2 chunk的size为0x151，然后freeorder2 chunk，再malloc(0x140)就可以让v5指向order2 chunk，这里要注意的是为了防止0x151的chunk和top chunk合并，以及绕过 Ptmalloc2 对next_chunk的检查，需要在0x151大小的chunk后面在使用溢出布置一个堆块；另一种做法是先通过释放order2 chunk，然后通过edit_order1修改free掉的order2 chunk的size为0x151，这样后面再malloc(0x140)也是可以得到order2 chunk；

本文payload用的是第一种方法。

submit函数将order1和order2中的内容连接起来放到v5中。因为通过溢出将order2 chunk size改为0x151，所以malloc的v5其实指向原先order2的地址，也就是v5和order2指向同一个地方。当先用strncat将order1中的内容放到v5中时，order2也就变成和v5一样的内容了，后面再用strncat连接order2时，就是v5和v5自己做连接：

因为后面要覆盖dest，所以需要计算好order1 chunk的长度，让后面一个order1 chunk刚好落在dest中，然后只需要在order1 chunk中写入格式化字符串即可。

所以order1_chunk中有0x74的空间可以用来写格式化字符串，写不满可以用其他非0字符填充。

漏洞利用：

结合上面的分析可以得出利用过程：

  1. 先edit_order1修改order2 chunk的size，并布置好堆，将格式化字符串写入order1_chunk，然后delete order2，再submit，利用格式化字符串泄露libc地址和栈地址并修改.fini_array[0]为main函数的地址；
  2. 根据泄露的libc地址和栈地址分别计算出libc基址和第二次运行main函数的返回地址，第二次运行main函数，重新布置堆空间，写入格式化字符串，修改返回地址为libc中execve的地址。为什么是execve，因为利用one_gadget工具可以找到libc中execve(/bin/sh)的代码，直接覆盖返回地址，不需要在寄存器或者栈上布置参数。

第一步：

通过调试可以得知用户输入的选项是格式化字符串的第12个参数：

所以submit('a' * 7 + p64(fini_array))将格式化字符串的第12个参数用5aaaaaaa填满，然后将fini_array的地址放到第13个参数上，再使用"%" + str(0xA39) + "c%13$hn"来修改fini_array中的内容为main函数的地址。%31$p是泄露[__libc_start_main+240]的地址，~是分隔符，方便后面接收泄露的地址。%28$p是泄露栈地址，因为在后面需要覆盖main函数的返回地址，返回地址存放到栈上，所以需要获取返回地址存放的栈地址才能利用格式化字符串去修改它，获取泄露的栈地址之后，通过提前调试计算返回地址所在的栈地址和泄露的栈地址之间的差值，就可以通过泄露的栈地址得到返回地址，这么做的原理是程序中的代码对栈的操作都是固定的，一旦程序的执行流程确定，那么在程序正常运行的情况下栈空间的变化是固定的。

为什么泄露的栈地址是格式化字符串的第28个参数？

这个不是固定的，也可以泄露其他值，有个原则是在第二运行main函数之后，泄露的栈地址还在栈上，这样的话计算其和返回地址之间的差值会方便一些。在第一步的payload中可以先不加',%28$p'，等后面调试的时候再确定泄露的栈地址。因为所有需要的值都要在第一次运行main函数的过程中泄露，所以栈地址需要在这里泄露。先不加',%28$p'调试，查看第二次运行main函数返回地址（第二次运行main函数exp可以先直接submit）：

返回地址所在的栈地址为0x7fffb2ea8668。通过对比第一次运行完main和第二次运行完main的栈空间，发现在栈地址0x7fffb2ea87b8 —▸ 0x7fffb2ea8850处，存放着栈地址0x7fffb2ea8850，而且在两次运行完main之后这个地址都在，所以它在第一、二次运行main的过程中不会变，这个栈地址可以用来泄露，调试可以发现这个地址正是之前第一次运行main函数是格式化字符串的第28个参数，所以可以使用%28$p来泄露。通过栈地址和返回地址可以计算出它们之间的差值为0x1e8。

exp后面edit_order1(payload)发送payload，delete_order2() free order2，submit('a' * 7 + p64(fini_array))将要修改的fini_array的地址布置到栈上。

Unlink

关于unlink：

unlink主要是为了实现两个相邻free chunk的合并。触发unlink的条件：

  1. 堆块free掉之后被放入到双向链表的bin中（unsorted bin，small bin，large bin），并且该free掉的堆块物理地址相邻的堆块也是出于free状态，触发unlink对这两个chunk进行合并（包括前向合并和后向合并）；先判断是否能前向合并（prev），然后再判断后向合并；
  2. malloc_consolidate()将fast bin中释放的chunk整理到unsorted_bin中；
  3. malloc()将unsorted中释放的chunk整理到small bin或者large bin中等。

一般用户可控的触发unlink的情况就是free chunk，通过堆溢出重新布置堆空间，将要free掉的chunk的前一个或者后一个chunk修改为free状态（一般是修改next_chunk的prev_size位），也可以直接fake一个chunk，在free 该chunk的时候就会触发unlink。

option function

  1. malloc chunk

    申请chunk，输入size，程序malloc size大小的chunk；所有的chunk指针都会存放到bss段的一个chunks数组中，最后输出chunk的数组索引；

  2. write_in_chunk

    向chunk中写入数据，先输入要写入的chunk的索引，然后输入要写入的数据大小，因为这里的count大小可控，所以存在堆溢出；fread函数每次向chunk_ptr中读入一个字节的数据，并返回已读入的数据大小；

    根据输入的索引free chunk，并将chunks数组中的chunk指针置为0；由于没有清空释放的chunk的内容，所以存在UAF；

  4. useless

    这个函数没有多大作用。

漏洞利用：

所有chunk的指针都存放到chunks数组中，因为unlink可以将变量的值减小0x18，所以想利用unlink将chunks数组中某一项chunks[x]中的值变为&chunks[x]-0x18，这样edit chunks[x]时，就是在修改chunks数组中的内容，通过可以修改chunks数组中的所有项。

利用过程：注意chunks数组的下标从1开始，chunks[0]没有使用

  1. 利用unlink修改chunks[1]=&chunks[1]-0x18，只需要伪造一个chunk，fd填入&chunks[1]-0x18，bk填入&chunks[1]-0x10，并且将伪造的chunk标记为free chunk（修改next_chunk的prev_in_use位），再free next_chunk（chunk 2），触发unlink，unlink之后chunks[1]就指向&chunks[1]-0x18；
  2. edit chunk1，即修改chunks数组的项，因为chunks[1]=&chunks[1]-0x18，修改chunks[0] = elf.got['free'], chunks[1] = elf.got['puts'], chunks[2] = elf.got['atoi']；
  3. edit chunk0，其实是修改chunks[0]指向的地方，因为chunks[0]指向free的got表，所以edit chunk0就是修改free的got表，将其修改为puts函数的plt表地址（puts@plt），这样调用free的时候就会跳到puts@plt执行，然后再跳到puts@got指向的puts函数执行；free chunk1，泄露puts@got；
  4. edit chunk2，修改chunks[2]，即修改atoi@got为system函数的地址，然后再将/bin/sh字符串输入给atoi函数（option = atoi(input);），即执行system(“/bin/sh”)。

问题：

  1. 为什么要写got表，不写plt表？

    got表可写，plt表不可写；

  2. 泄露是泄露got表地址，覆盖是将got表覆盖为plt表的地址？

    泄露是got表地址：在第二次调用函数时，got表中存放的是函数的地址；程序会jmp到got表指向的地址处执行：jmp *0xaaaaaaa；

    覆盖是plt表的地址：plt表中是指向一条jmp指令，直接jmp到got表指向的地址执行，got表里面就是一个地址，里面不是指令，直接jmp过去不能执行，需要jmp *取里面的指令地址跳转过去才能执行，plt表中存放的是指令的地址，跳转过去就能直接执行。

具体过程：

因为程序没有使用setbuf设置输入输出的缓冲区，当程序有输入输出的时候，会自己malloc堆块来作为输入输出缓冲区。程序在调用fgets和puts时，会分别申请一个0x410大小的chunk。

为了在后续的利用中不受缓冲区chunk的影响，先让程序把缓冲区申请好，意思就是执行一遍fgets和puts函数，一开始运行程序的时候，随便输入一个数字选项（1-4之外的），让while循环走一遍，while循环一遍之后，fgets函数和puts函数都执行一遍了，就申请好了上面两个缓冲区的chunk。

申请chunk1和chunk2，并修改chunk1，因为要修改chunks[1] = &chunks[1] - 0x18，通过前面的分析，要修改chunks[1]的值，需要chunks[1]指向chunk的头部（prev_size的位置），所以需要编辑chunk1的时候在chunks[1]指针处伪造一个chunk，修改fd为&chunks[1]-0x18，bk为&chunks[1]-0x10；然后通过堆溢出修改next_chunk的prev_size和prev_inuse位，将chunk1中伪造的chunk标记为free chunk，然后free chunk2，触发unlink。

unlink之后就可以通过edit chunk1来修改chunks数组的各个项，这里修改chunks[0] = elf.got['free'], chunks[1] = elf.got['puts'], chunks[2] = elf.got['atoi']；然后再修改chunks[0]，也就是修改free的got表中的内容，将其修改为puts@plt，再free chunk1，就相当于执行puts(puts@got)，泄露puts函数的地址。

通过泄露的puts函数的地址可以计算出libc基址，然后edit chunk2，将atoi函数的got表内容修改为system函数的地址，然后再send “/bin/sh”字符串，这样执行atoi(input)就是在执行system(“/bin/sh”)。
